FASE 1 — MVP FLUTTER COMPLETO (SIN BACKEND, SIN CUENTAS)
Estado actual: COMPLETADA (2026-06-14)
Duración estimada: 1–2 sprints (pulido + gaps)
================================================================================

Objetivo:
  App funcional multiplataforma con toda la pedagogía en cliente y persistencia
  local. Producto usable para entrenamiento real diario.

Entregables funcionales [M]:
  - Modos: una nota, intervalo, acorde, aleatorio, velocidad
  - SRS híbrido operativo con guardado local
  - Cluster de limpieza entre rondas
  - Piano interactivo + input texto
  - Pantalla estadísticas + recomendaciones
  - Variación octavas + timbres aleatorios

Entregables técnicos [M]:
  - Suite test: unit + integration + widget + e2e verde
  - ProgressRepository con migración formato antiguo (si aplica)
  - AudioPlayerService estable en Windows + Web + al menos un móvil

Entregables técnicos [S]:
  - Go Router para navegación (reemplazar Navigator imperativo)
  - Tema visual coherente (Material 3; identidad musical propia)
  - Modo oscuro

Criterios de aceptación (DoD):
  - Usuario completa 10 rondas, cierra app, reabre: progreso SRS conservado
  - flutter test → 0 failures
  - Audio audible en dispositivo objetivo principal del desarrollador
  - Flujo home → juego → resultado → cluster → siguiente ronda sin crashes

Tareas pendientes típicas (auditar contra código):
  1. [x] Confirmar audio_player_service import real de flutter_soloud
  2. [x] Probar flutter run -d chrome (web) — manual
  3. [ ] Probar flutter run en Android/iOS si es target — manual
  4. [x] Persistencia cross-platform via SharedPreferences
  5. [x] Onboarding pedagogico (SRS, octavas, cluster)
  6. [x] Go Router + tema claro/oscuro

FASE 2 — CALIDAD, UX Y PRE-LANZAMIENTO
Estado actual: EN PROGRESO (2026-06-14)
Duración estimada: 2 sprints
Depende de: Fase 1 DoD
================================================================================

Objetivo:
  Producto presentable públicamente: estable, accesible, medible, desplegable.

Entregables [M]:
  - CI GitHub Actions: analyze + test en cada push/PR
  - Build web automatizado (artefacto deployable)
  - Sentry o equivalente para crashes [S]

Entregables UX [S]:
  - Design system mínimo (tokens color, espaciado 8px, tipografía)
  - Tap targets piano ≥ 48dp en móvil
  - Estados vacío/error/loading en pantallas clave
  - Accesibilidad básica: Semantics en piano, contraste WCAG AA

Entregables producto [S]:
  - Analytics anónimo (PostHog/Firebase): sesiones, modos usados, retención
  - Pantalla "Acerca de" con explicación pedagógica

Entregables [C]:
  - PWA: manifest + service worker para web offline parcial
  - Localización preparada (i18n) aunque solo español al inicio

Criterios de aceptación (DoD):
  - Pipeline CI verde en repo remoto
  - Web desplegada en URL de preview (Vercel/Firebase)
  - Crash rate 0 en smoke test manual de 30 min
  - Lighthouse accesibilidad ≥ 90 en web [S]

Principios .cursorrules aplicados:
  - Continuous Integration (merge frecuente, CI obligatorio)
  - Clean Code: funciones ≤ 20 líneas donde sea razonable en Dart
  - Regla Boy Scout en archivos tocados

FASE 4 — CUENTAS Y SYNC (SUPABASE)
Duración estimada: 2–3 sprints
Depende de: Fase 3 DoD + tracción o necesidad explícita de multi-dispositivo
================================================================================

Objetivo:
  Usuario opcional puede crear cuenta y sincronizar progreso SRS entre dispositivos.

Entregables [M]:
  - Supabase project: Auth (email/OAuth), Postgres, RLS
  - Esquema: users, note_progress (JSON o filas por nota), sessions_metadata
  - ProgressRepository remoto implementando misma interface que local
  - Flujo: "Continuar sin cuenta" vs "Vincular / Crear cuenta"
  - Migración: importar progreso local al registrarse

Entregables [S]:
  - Sync incremental offline-first (cola local → push al reconectar)
  - Recuperación de contraseña, verificación email

Entregables [W por ahora]:
  - Cualquier backend propio en Python
  - GraphQL

Criterios de aceptación (DoD):
  - Mismo usuario en web + móvil ve mismo progreso SRS tras sync
  - RLS impide leer progreso de otros usuarios
  - App funciona offline; sync no bloquea juego

Arquitectura:
  Flutter → supabase_flutter → Postgres (RLS)
  Audio y SRS siguen en cliente; servidor solo persiste estado.

FASE 5 — MONETIZACIÓN (MODELO DE NEGOCIO)
Duración estimada: 2 sprints
Depende de: Fase 4 DoD (cuentas necesarias para sync Pro)
================================================================================

Objetivo:
  Plan freemium operativo con pagos reales.

Entregables [M]:
  - Definición planes: Free vs Pro (qué modos/features gating)
  - RevenueCat integrado (iOS + Android)
  - Stripe Checkout o Customer Portal (web)
  - Paywall UI en Flutter
  - Entitlements reflejados en Supabase (cache del estado de suscripción)

Entregables [S]:
  - Trial 7–14 días
  - Restaurar compras
  - Pantalla gestión suscripción

Criterios de aceptación (DoD):
  - Compra sandbox exitosa en iOS/Android + web test mode Stripe
  - Usuario Free no accede a features Pro; usuario Pro sí
  - Restaurar compras funciona en segundo dispositivo

Reglas negocio sugeridas (ajustables):
  Free : una nota, intervalo, SRS local, stats básicas
  Pro  : acorde, aleatorio, velocidad, stats avanzadas, sync, export [C]

FASE 6 — ESCALA, OPERACIÓN Y EVOLUCIÓN
Duración estimada: continua
Depende de: Fase 5 si hay negocio; Fase 3 si permanece gratuito
================================================================================

Objetivo:
  Producto sostenible, observable y evolutivo.

Entregables [S]:
  - Dashboard métricas: retención D1/D7, conversión Free→Pro, modos más usados
  - Alertas Sentry + uptime web
  - Backups Supabase automatizados
  - Dependabot / renovate para deps Flutter

Entregables [C]:
  - Modo canto/tarareo (respuesta por micrófono)
  - Solfeo Do/Re/Mi además de C/D/E
  - Notificaciones push "tienes notas vencidas para repasar"
  - Export CSV del progreso

Métricas externas (adaptadas de .cursorrules):
  - Crash rate mobile ≤ 0.1% sesiones
  - CSAT ≥ 4.2/5 (encuesta in-app ocasional)
  - Retención D7 ≥ objetivo definido por producto

Deuda técnica:
  - Revisión trimestral; ratio deuda ≤ 5% del backlog
